Engineering Decisions

Why These Technologies

Every architectural choice was driven by the specific requirements of clinical decision support â€” not convention.

Neo4j over Vector Databases

Drug interactions are fundamentally relational, not semantic.

Vector embeddings excel at semantic similarity â€” finding documents that "mean" similar things. But drug interactions require structural reasoning: Drug A inhibits Enzyme CYP3A4, which metabolizes Drug B, leading to toxic accumulation.

This is a graph traversal problem. Neo4j allows multi-hop queries that follow these biochemical pathways. A vector database would surface "similar" drugs but miss the causal chain that creates the actual clinical risk.

Trade-off acknowledged: Graph databases require explicit schema design and don't handle unstructured queries as gracefully. I mitigated this with the Text-to-Cypher layer.

Local LLM (Ollama) over Cloud APIs

Healthcare data demands privacy-first architecture.

Sending patient medication lists to external APIs â€” even for query generation â€” creates compliance and privacy concerns. Ollama with Llama 3.2 runs entirely on-device, ensuring patient context never leaves the local environment.

The 3B parameter model was specifically chosen after benchmarking: it provides sufficient reasoning capability for Cypher generation while fitting in 8GB VRAM with Metal acceleration on macOS.

Trade-off acknowledged: Smaller models have lower accuracy ceiling than GPT-4. The self-healing retry mechanism compensates â€” if the first query fails, the error feedback loop typically recovers within 3 attempts.

Synthetic Data Generation

Real patient data is inaccessible, but realistic testing is non-negotiable.

Clinical decision support systems fail in production when they encounter edge cases: elderly patients on 12+ medications, rare drug combinations, or unusual comorbidity patterns. Testing against synthetic "happy path" data masks these failures.

The generator creates correlated patient profiles â€” elderly patients probabilistically have more chronic conditions, which leads to more medications, which correlates with reduced organ function. This mimics real population distributions.

Trade-off acknowledged: Synthetic data can't capture true clinical complexity. The 752 profiles serve as a stress test, not a substitute for real-world validation.

PyVis for Visualization

Interaction networks are inherently visual â€” tables fail to communicate risk topology.

When a patient takes 8 medications, there are potentially 28 pairwise interactions. A table listing these is cognitively overwhelming. A force-directed graph immediately reveals the "hub" drugs that interact with everything, and clusters of related risks.

PyVis was chosen over D3.js for development velocity â€” it integrates directly with Python/Streamlit without requiring a JavaScript build pipeline. The interactive physics simulation runs client-side.

Trade-off acknowledged: PyVis has limited customization compared to D3. For production, I'd consider migrating to a React-based graph library for finer control over styling and interaction.
